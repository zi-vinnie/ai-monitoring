# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

A parental screen-monitoring system with three components, laid out as a monorepo (one directory per component). `windows-agent/`, `server/`, and `classifier/` all exist. The classifier component also hosts the daily email-report script (`send-report`).

1. **Windows agent** (`windows-agent/`, see `windows-agent/README.md`) — Python + FastAPI + `mss`, built with `uv` (src layout, `pyproject.toml` + `uv.lock`). Runs as a hidden background service on the monitored Windows machine. Exposes a `/screenshot` endpoint, protected by an API key (`X-API-Key` header, checked with `secrets.compare_digest` for a constant-time comparison), that captures **only the monitor currently holding the focused window** and returns it as a single base64 PNG. Uvicorn binds to `0.0.0.0:8000` (all interfaces) so the server can reach it over the LAN — inbound access should be scoped to the server's IP via a Windows Firewall rule (see `windows-agent/README.md`), since the API key is currently the only application-layer protection. Active-window/monitor detection uses `pywin32` (`GetForegroundWindow` + `MonitorFromWindow`), declared with an `; sys_platform == 'win32'` marker in `pyproject.toml` so the project still resolves on non-Windows dev machines — the win32-specific code in `windows_agent/active_monitor.py` is import-guarded and only raises if actually called off-Windows. `windows_agent/capture.py` matches the win32 monitor rect to mss's monitor index via `match_monitor_index` (a pure function, unit-testable without a real display or Windows). Because mss/GDI capture returns an all-black frame for exclusive-fullscreen DirectX/OpenGL games (Rocket League etc.), `capture.py` detects a black frame (`is_mostly_black`, also pure/unit-testable) and re-captures that monitor via the Desktop Duplication API in `windows_agent/capture_dxgi.py` (`dxcam`, declared with `; sys_platform == 'win32'` and import-guarded). The DXGI path is fallback-only — it runs solely when the mss frame is black and returns `None` on any failure, so normal desktop captures are unchanged and behaviour can't regress. It matches the active monitor to a `dxcam` output by HMONITOR (`get_active_monitor_handle`). Entry point: `uv run windows-agent`.
2. **Ubuntu server** (`server/`, see `server/README.md`) — plain Python, no n8n, built with `uv` (src layout). A `poll-screenshots` script (invoked on a schedule via cron or a systemd timer, not an internal loop) calls the Windows agent's `/screenshot` endpoint every 5 minutes, saves the single PNG under `data/screenshots/<date>/`, and records metadata (timestamp, monitor index, focused window title, file path, label) in SQLite (`server/src/server/db.py`). When a poll fails it logs the reason and records a row in the `agent_status` table (`server/src/server/poll.py`): `unreachable` for a connection error/timeout (machine off, asleep, or agent stopped) and `error` for any other HTTP failure. (These rows are kept for diagnostics but are no longer surfaced in the daily report — a failed poll only means the PC was off or offline, which isn't actionable.) Config is via env vars (`AGENT_URL`, `AGENT_API_KEY`, `SCREENSHOT_DIR`, `DB_PATH`) loaded from `.env` — see `server/.env.example`. Auto-deletion of images after a few days (retaining labels longer) is not yet implemented.
3. **Local AI classifier** (`classifier/`, see `classifier/README.md`) — plain Python built with `uv` (src layout), sharing the server's SQLite DB (`DB_PATH` points at it; the classifier reads rows and fills the `label` column but never creates tables — `server` owns the schema). Two single-shot, once-daily entry points (systemd timers, no internal loop): `classify-screenshots` and `send-report`. Both accept `--date` (`YYYY-MM-DD`, `today`, or `yesterday`); the intended schedule covers the *previous* day — classify ~03:00, report ~09:00, both with `--date yesterday` (the gap doubles as a retry window for rows left unlabeled by an Ollama outage). **`classify-screenshots`** reads the day's `label IS NULL` screenshots and, for each, first checks a small hard-coded exact-title map (`label_for_title` in `categories.py`, case-insensitive) that short-circuits titles the title alone settles: unambiguous game clients (e.g. `Overwatch`, `Rocket League (64-bit, DX11, Cooked)`) go straight to `gaming`, and the bare desktop — the Windows `Program Manager` shell or a null/empty window title — goes straight to `idle`; both skip the image encode and Ollama call, which is cheaper and (for the games) sidesteps the exclusive-fullscreen black/wallpaper-frame problem; otherwise it base64-encodes the PNG and calls a local **Ollama** vision model (`OLLAMA_MODEL`, default `minicpm-v4.6`) via `/api/generate` at `temperature 0`, passing the focused window title as a hint and Ollama's structured-output `format` (a JSON schema that asks for a one-sentence `screen_content` description *before* a `label` constrained to the six categories — the description-first ordering is a deliberate cheap reasoning step for the small vision model) so the model can only return a valid label. Labels each into one of: `productive` (broad — trading/investing, coding, writing/research/learning, email/calendar/meetings, applications, documents, AI assistants, creative tools, system settings, file management, and educational/how-to/finance videos), `gaming` (playing yourself, incl. launchers; watching others play counts as video_entertainment), `video_entertainment` (video watched for fun; educational/finance videos count as productive instead), `social_media` (feeds and messaging), `browsing` (general web browsing), and `idle` (machine on but unused — bare desktop, lock/login screen, blank frame; also the model's fallback when no activity is identifiable). The prompt biases toward `productive` as the default category when unsure, and reserves `idle` for genuinely no-activity frames. Since only the focused monitor is captured, there's no cross-monitor reconciliation/priority step — one image in, one label out. Per-image failures (Ollama down/timeout, missing image file, unparseable response) are logged and skipped, leaving the row unlabeled for a later run to retry, rather than aborting the batch. Category list + prompt + label parsing live in `classifier/src/classifier/categories.py`; keep the taxonomy in sync there and here. "A day" is defined by `REPORT_TZ` (local calendar day → UTC bounds via `timeframe.py`, since `captured_at` is stored UTC).
4. **Email summary** (`classifier/`, `send-report` entry point) — plain Python (stdlib `smtplib`, no n8n, no third-party email SDK). The headline visual is a **day-timeline PNG** (`chart.py` `render_timeline`, matplotlib `broken_barh`): a single horizontal track spanning the local day 00:00→24:00, coloured by activity at each time (fixed per-category hues, same palette across the report). `idle` samples draw as a muted-grey block (machine on but unused); only a genuinely missed poll leaves the empty track colour, so grey vs empty distinguishes idle-at-desktop from machine-off/gap. The timeline is built in `timeline.py` (`build_timeline`): ordered `(captured_at, label)` samples (`db.fetch_labeled_events`) are each converted to the local `REPORT_TZ` minute-of-day and given the span until the next sample, capped at one `POLL_INTERVAL_MINUTES`; consecutive same-category samples merge into one block (e.g. three back-to-back Rocket League frames → one gaming block) while a missed poll opens a visible gap. Alongside it the email carries headline stats (screen time, productive share) and a per-category breakdown table, both from `aggregate.py` (`count × POLL_INTERVAL_MINUTES`, each screenshot standing in for one poll interval). Headline screen time and the productive-share denominator are *active* minutes only — `idle` is excluded from those totals (`active_minutes`) and broken out as its own breakdown row (with no share %), since it's machine-on-but-unused time rather than active use. Emails a plain-text + HTML summary with the chart embedded inline (`cid:`) and attached, to every address in `EMAIL_TO` (comma-separated), over SMTP (STARTTLS by default, `SMTP_SSL=true` for implicit-TLS port 465; both paths verify the server certificate via `ssl.create_default_context()`). Falls back to a "no activity recorded" notice when nothing was labeled. Email was chosen over WhatsApp deliberately: no WhatsApp 24-hour session window / template pre-approval, and the chart rides along as bytes (SMTP multipart) with no need to host it at a public URL.

n8n was considered for the server-side polling/aggregation/notification steps but rejected in favor of plain Python: simpler to test, version-control, and reason about for logic this straightforward. Capturing all monitors and stitching/reconciling them was also considered and rejected in favor of active-window detection — it's simpler (no reconciliation logic), cheaper (half the storage and classification calls with two monitors), and more accurate (captures literal focus instead of inferring which monitor's activity "wins").

### Windows agent `/screenshot` contract

```json
{
  "captured_at": "2026-07-01T13:20:00+00:00",
  "monitor_index": 1,
  "window_title": "Instagram - Google Chrome",
  "png_base64": "..."
}
```
Request: `GET /screenshot` with header `X-API-Key: <key>`. `window_title` is nullable (e.g. no focused window). If this contract changes, update both `windows_agent/app.py`/`windows_agent/capture.py` and `server/src/server/fetch.py`/`server/src/server/poll.py` together.

## Security

- The Windows agent's `/screenshot` endpoint is API-key protected — never remove or weaken this auth, and never log or print the API key.
- Traffic between `server` and `windows-agent` is currently plain HTTP, so the API key and screenshot bytes are sent in cleartext over the LAN — anyone already positioned on the network (or on the router) can sniff them. Restricting inbound access to the server's IP (firewall rule) narrows who can *initiate* requests but does not fix this; putting the link behind a VPN/overlay (e.g. Tailscale/WireGuard) or adding TLS would close it. Flag this if asked about production-hardening the transport.
- This system captures screenshots of a monitored machine and is intended strictly for authorized parental monitoring of a household device. Do not extend it toward covert surveillance of adults without consent, or add features (remote install, hidden persistence beyond the stated hidden-service design, keylogging, etc.) that would turn it into general-purpose stalkerware.
- Image retention/deletion windows (a few days for images, longer for labels) are a deliberate privacy tradeoff — don't casually extend retention when touching the cleanup job.
